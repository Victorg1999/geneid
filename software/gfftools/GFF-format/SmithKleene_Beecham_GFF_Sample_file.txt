# GFF sample record
#
#  Official GFF specification -> http://www.sanger.ac.uk/Software/GFF/
#
#       Definition
#  
#  Fields are: <seqname> <source> <feature> <start> <end> <score> <strand> <frame> [group]
#  
#  <seqname>
#       The name of the sequence. Having an explicit sequence name allows a feature file to be
#       prepared for a data set of multiple sequences. Normally the seqname will be the
#       identifier of the sequence in an accompanying fasta format file. An alternative is that
#       'seqname' is the identifier for a sequence in a public database, such as an
#       EMBL/Genbank/DDBJ accession number. Which is the case, and which file or database to use,
#       should be explained in accompanying information.
#  <source>
#       The source of this feature. This field will normally be used to indicate the program
#       making the prediction, or if it comes from public database annotation, or is
#       experimentally verified, etc.
#  <feature>
#       The feature type name. We hope to suggest a standard set of features, to facilitate
#       import/export, comparison etc.. Of course, people are free to define new ones as needed.
#       For example, Genie splice detectors account for a region of DNA, and multiple detectors
#       may be available for the same site, as shown above.
#  <start>, <end>
#       Integers. <start> must be less than or equal to <end>. Sequence numbering starts at 1, so
#       these numbers should be between 1 and the length of the relevant sequence, inclusive.
#       (Version 2 change: version 2 condones values of <start> and <end> that extend outside the
#       reference sequence. This is often more natural when dumping from acedb, rather than
#       clipping. It means that some software using the files may need to clip for itself.)
#  <score>
#       A floating point value. When there is no score (i.e. for a sensor that just records the
#       possible presence of a signal, as for the EMBL features above) you should use '.'.
#       (Version 2 change: in version 1 of GFF you had to write 0 in such circumstances.)
#  <strand>
#       One of '+', '-' or '.'. '.' should be used when strand is not relevant, e.g. for
#       dinucleotide repeats.
#  <frame>
#       One of '0', '1', '2' or '.'. '0' indicates that the specified region is in frame, i.e.
#       that its first base corresponds to the first base of a codon. '1' indicates that there is
#       one extra base, i.e. that the second base of the region corresponds to the first base of
#       a codon, and '2' means that the third base of the region is the first base of a codon. If
#       the strand is '-', then the first base of the region is value of <end>, because the
#       corresponding coding region will run from <end> to <start> on the reverse strand. As with
#       <strand>, if the frame is not relevant then set <frame> to '.'. It has been pointed out
#       that "phase" might be a better descriptor than "frame" for this field.
#  [group]
#       An optional string-valued field that can be used as a name to group together a set of
#       records. Typical uses might be to group the introns and exons in one gene prediction (or
#       experimentally verified gene structure), or to group multiple regions of match to another
#       sequence, such as an EST or a protein. See below for examples.
#       Version 2 change: In version 2, the optional [group] field on the line must have an tag
#       value structure following the syntax used within objects in a .ace file, flattened onto
#       one line by semicolon separators. Tags must be standard identifiers
#       ([A-Za-z][A-Za-z0-9_]*). Free text values must be quoted with double quotes. Note: all
#        non-printing characters in such free text value strings (e.g. newlines, tabs, control
#       characters, etc) must be explicitly represented by their C (UNIX) style backslash-escaped
#       representation (e.g. newlines as '\n', tabs as '\t'). As in ACEDB, multiple values can
#       follow a specific tag. The aim is to establish consistent use of particular tags,
#        corresponding to an underlying implied ACEDB model if you want to think that way (but
#       acedb is not required). Examples of these would be:
#  
#       seq1     BLASTX  similarity   101  235 87.1 + 0 Target "HBA_HUMAN" 11 55 ; E_value 0.0003
#  
#       dJ102G20 GD_mRNA coding_exon 7105 7201   .  - 2 Sequence "dJ102G20.C1.1"
# 
#  All strings (i.e. values of the <seqname>, <source> or <feature> fields) should be under 256
#  characters long, and should not include whitespace. The whole line should be under 32k long. A
#  character limit is not very desirable, but helps write parsers in some languages. The slightly
#  silly 32k limit is to allow plenty of space for comments/extra data. Version 2 change: field
#  and line size limitations are removed; however, fields (except the optional [group] field
#  above) must still not include whitespace.
#  
#  All of the above described fields should be separated by TAB characters ('\t'). Version 2
#  note: previous Version 2 permission to use arbitrary whitespace as field delimiters is now
#  revoked! (99/02/26)
# 
#
#  SmithKline-Beecham
#  GFF Extensions: 
#     1.  Group field (9th field) is formalized and required just like 
#            the frame, score, etc. fields
#
#     2.  Comment field (10th field):  Name value; Name1 value1 value1 value1; 
#            Parseable comments fields have the "name values;" format.
#            Can't mix parseable comments with non-parseable comments.
#    
#     3.  Score=* indicates a continuous feature field.  The values for the Score will be
#            listed as indicated in #2 above.  Score X Y Z U V W; Step 3;
#            Other than the * in the score field, this will be a normal feature with
#            a set of scores in the comment field and optional step keyword if the
#            score is associated with every nth base.  All features with the same group
#            field are suggested to be placed upon the same graph.  Optional Types will be
#            indicated to detail how to display the graph.
#
#     4.  Blast or multiple sequence alignments are possible to indicate through the use 
#            comment field descriptors Seq and Map.  Seq ATGACT...; Align (1.10/30.39) (11.15/53.57);
#            Example indicates the sequence to be aligned against the primary sequence and
#            that this is a gapped alignment where the primary sequence bases=1-15 are aligned against 
#            two pieces of the aligned sequence.
#            
#     5.  Continuation line:  a "\" (backslash) placed at the end of a line indicates that one is to
#            collect the following line with the current line such that the spaces prior to the first
#            non-space character on the following line are removed and the following line attached 
#            exactly at the backslash (this is to remove the possibility of inserting a space inside of
#            a sequence).
#
#     6.  Additional extension items are available to enrich the context of the features in the sequence 
#            viewer and are less noteworthy than the prior main extensions.
#            
#

#  Controlled Feature Vocabulary (case ignored)
#       Gene    5' UTR to 3'UTR - encompasses every base between these two points
#       mRNA    5' UTR to 3' UTR after removing introns
#       CDS     start to stop - only coding sequence - no introns allowed
#               only way to include this as a single feature is if only one exon is found.
#               Can also build from ExonB to ExonE.  This is a useful thing to be able to
#               view as a construct.  Link exons starting from the start codon to the stop codon.
#       Start   start codon
#       Exon    Exon
#       Exon_Alt        Alternatively spliced exon (use glyph to stand out yet still relate to gene)
#       Intron  non-coding sequence internal to gene that is spliced out
#       5'UTR   5' UTR  
#       3'UTR   3' UTR
#       Promoter|Enhancer|TSS|TATA_Box|TFBP_Element|Regulatory_Reg      Regulatory regions
#               These identifiers all refer to regulatory regions and should have the same glyph
#               TSS=Transcription start site, TFBP=regulatory protein region
#       SNP     Single Nucleotide Polymorphism
#       CpG     CpG islands
#       EST     see blastn - uses same format
#       rRNA    ribosomal RNA
#       tRNA    t-RNA
#       STS     sequence tagged site
#       PolyA_signal    PolyA signal
#       PolyA_site      PolyA site
#       Repeat_region   Repetitive sequence region
#       blastx|blastp|tblastx|tblastn=protein similarity    EST|blastn=nucleotide similarity
#               protein similarities are based upon nucleotide numbering for the query (GFF nucleotide
#               seq) and peptide numbering for the subject sequence.  The protein sequence will need 
#               to be displayed in the center of the nucleotide sequence triplets it corresponds to.
#               Score,Prob,Identity,Match are keywords for the appropriate values from the Blast output
#               TargetSeq=subject HSP sequence; Desc subject description;  Align X;
#                       in free field comments section
#               Align=(1.10/1.10) (13.40/11.38) (41.50/45.64) -> for describing gaps in query seq
#                       describes a gap in the query sequence of 2 bases at location 11-12 and a gap
#                       in the subject sequence at 39-44.  The / separates the Query/Subject sequence
#                       seqments.  The parenthesis capture non-gapped sub-sequences.
#
